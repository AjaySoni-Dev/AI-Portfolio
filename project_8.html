Vision-X: Low-Power Assistive Smart Spectacle

A mobile-centric Edge AI system for assistive navigation using two-part architecture: Wearable sensor node + Android AI decision system.

View System Design Download Documentation

System Partitioning Overview

🟦 PART-A

ANDROID MOBILE EDGE AI & DECISION SYSTEM

The brain of the system. All intelligence, reasoning, AI inference, and user interaction happen here.

✅ COMPLETED

🟨 PART-B

WEARABLE RASPBERRY PI SENSOR & DATA NODE

The eyes + distance sensing layer, deliberately kept dumb, safe, and low-power.

⏳ PENDING (HARDWARE + LOW-LEVEL SOFTWARE)
ARCHITECTURAL RULE: The wearable never makes decisions. The mobile never trusts raw data blindly.

The entire Vision-X system is intentionally divided into two independent but cooperative subsystems. This modular approach ensures safety, scalability, and maintainability.

Part-A (Android Mobile): Acts as the Edge AI compute unit, decision authority, alert generation system, and user interaction interface. It receives raw sensory data, transforms it into meaning, and outputs only actionable spoken alerts.

Part-B (Raspberry Pi Wearable): Acts as a sensor aggregator, data transport node, and fail-safe wearable device. It does NOT run AI, make decisions, or generate speech - ensuring low power consumption and safety.

🔗 System Integration

Frame Receiver Module

Receives frames in real-time from phone camera or external stream (JPEG/MJPEG from Pi). Handles frame buffering safely and ensures consistent frame rate for inference.

Image Pre-Processing Module

Converts frames to RGB, resizes to YOLO input resolution, normalizes, and memory-optimizes. Reduces inference latency and prevents crashes.

Edge AI Object Detection

YOLOv10-n (nano) model runs entirely on-device. No cloud or internet required. Outputs object class, bounding box, and confidence score.

Post-Processing & Confidence Filtering

Removes low-confidence detections, refines bounding boxes, filters noise. Prevents flickering alerts and false positives.

Sensor Fusion Engine

Combines vision-based detections (from YOLO) with distance data (from ultrasonic sensor) and temporal context. Determines object relevance and danger.

Decision Logic Engine (CRITICAL)

Generates spoken alert ONLY IF ALL conditions are true: object confidence > threshold, distance < safety limit, alert cooldown expired.

Text-to-Speech & Audio Output

Offline Android TTS output through bone-conduction earphones. Ears remain open to ambient sound - no audio generated on Raspberry Pi.

User Interface & Settings

Mode selection (Phone/External Stream), minimal accessible UI designed specifically for visually impaired users.

Hardware Components

Raspberry Pi Zero 2 W, Camera Module, HC-SR04 Ultrasonic Sensor, voltage divider for safety, 18650 Li-ion battery with TP4056 charging & protection.

Camera Module

Captures raw frames only. No processing or inference happens on the wearable device.

Ultrasonic Distance Module

Measures obstacle distance in cm. Pure sensing with no logic - converts echo time to distance.

Timestamping Unit

Adds precise timestamps to sensor readings for synchronization on mobile side.

Wi-Fi Hotspot Module

Pi creates its own hotspot. Mobile connects directly - no external router required.

Passive Fail-Safe Controller

MOST IMPORTANT: Checks if device connected to hotspot. If NO: stops all streams, enters low-power idle. Ensures phone is always in control.

Power & Energy Strategy

No AI computation, no audio output, sensors active only when mobile connected. ~2.5–3 hours prototype runtime with easily scalable battery.

Data Transport

ONE-WAY ONLY: Wearable → Mobile. Sends JPEG/MJPEG frames, ultrasonic distance (cm), timestamp. No commands accepted from mobile.

System Workflow

Raspberry Pi (Sense + Stream) → Android Mobile (AI + Decision + Speech) → User (Minimal, meaningful alerts)

Part-A Guarantees

Fully offline, privacy-preserving, low latency, scalable to better models, centralized intelligence.

Part-B Restrictions

Does NOT: Run AI, Run YOLO, Generate speech, Make decisions. Hardware failures don't cause unsafe behavior.

Why This Design Is Strong

Modular development, easy debugging, hardware-software separation, mobile hardware evolves → system improves automatically.

Part-A Flowchart

Start → Select Mode → Receive Frame → Preprocess → YOLOv10 → Post-Process → Receive Distance → Sensor Fusion → Decision → Alert?

Part-B Flowchart

Power ON → Initialize Sensors → Create Hotspot → Connected? → Capture Data → Timestamp → Stream → Repeat

Mobile-Centric Advantage

Leverages powerful mobile processors for AI, automatic updates via Play Store, familiar user interface, no custom hardware development needed.

Academic & Real-World Viability

Designed for both academic research (modular, well-documented) and real-world deployment (safe, low-power, user-friendly).

Technical Specifications

Part-A: Android System

The complete Edge AI decision system running on Android mobile:

  • Framework: Android (Kotlin/Java)
  • AI Model: YOLOv10-n (nano)
  • Inference: On-device, no cloud
  • TTS: Offline Android Text-to-Speech
  • Privacy: 100% local processing
  • Interface: Accessible for visually impaired

Part-B: Wearable Hardware

Low-power sensor aggregation node:

  • Core: Raspberry Pi Zero 2 W
  • Sensors: Camera Module + HC-SR04 Ultrasonic
  • Power: 18650 Li-ion (3.7V) with TP4056
  • Safety: Voltage divider (5V → 3.3V)
  • Connectivity: Wi-Fi Hotspot
  • Runtime: ~2.5-3 hours (prototype)

AI & Performance

Edge AI capabilities and metrics:

  • Model: YOLOv10-n (optimized for mobile)
  • FPS: 15-20 on mid-range Android
  • Accuracy: COCO dataset benchmark
  • Classes: 80+ object categories
  • Latency: < 100ms end-to-end
  • Memory: < 200MB model size

Innovation & Advantages

Key innovative aspects:

  • Two-part safety architecture
  • Mobile-centric Edge AI
  • Fail-safe wearable design
  • No cloud dependency
  • Bone-conduction audio
  • Modular & scalable
